- O `marcos-agent` usa memória persistente sem tratar memória como fonte oficial da verdade. -> comprovado: `createMarcosMemory()` passou a configurar `Memory` com `lastMessages`, `workingMemory` e política explícita; `marcos-agent.ts` injeta essa memória no agente, enquanto `marcosMemoryPolicy.knowledgePrecedence` fixa `knowledge` como fonte oficial. RF-12, RF-13, RF-16.
- O assembler de contexto carrega apenas o necessário e consegue explicar a origem do contexto usado. -> comprovado: `buildMarcosExecutionContext()` seleciona fatos relevantes, capabilities, workflows e chunks RAG mínimos, preservando a ordem `constitution -> system-prompt -> memory -> capability -> workflow -> rag` e emitindo `sourceTrace`. RF-13, RF-16, RF-24.
- A trilha de auditoria cobre workflow, capabilities, approval status e correlation ID. -> comprovado: `bootstrapMarcosExecutionAuditSchema()` cria `agents.marcos_execution_audit`; `recordMarcosExecutionAudit()` persiste `workflowId`, `capabilityIds`, `approvalStatus`, `correlationId` e fatos permitidos; `MastraTelegramAgentRuntime` já registra a execução transitória do Marcos. RF-18, RF-19.
- Existe proteção explícita contra gravação de dados proibidos. -> comprovado: `sanitizeMarcosMemoryFacts()` rejeita segredos, campos sensíveis, valores extensos e replay de documentos oficiais completos antes de memória persistente ou auditoria; os testes validam os bloqueios. RF-14, RF-15.
